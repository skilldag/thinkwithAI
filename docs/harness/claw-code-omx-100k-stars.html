Claw Code：如何在24小时内获得10万Star #

当开源项目的代码被一夜泄露，如何用一夜时间重写出超过100k Star的开源框架？

问题背景 #

2026年3月31日，一个来自 Anthropic 的热门工具 Claude Code 意外泄露。与以往被 DMCA 下架的类似项目不同，开发者 instructkr 采取了完全不同的策略：净室重写（Clean Room Rewrite）。

我遇到了一个困惑：为什么很多直接复制粘贴的仓库很快就被删除，而 Claw Code 不仅没有消失，反而创造了 GitHub 历史上最快达成 100k Stars 的记录？

探究过程 #

第一次尝试：直接使用 Claude Code #

最初我的想法是：既然代码泄露了，直接用官方版本不就行了吗？

  • 下载了 Claude Code 的 leaked 版本
  • 尝试配置和运行
  • 遇到问题：隐私风险，不想将编码任务上传到官方服务器

追问一：法律风险问题 #

既然官方版本有争议，有没有替代方案？

在网上搜索后发现：

  • 大部分仓库迅速被 DMCA 下架
  • 直接复制粘贴代码存在法律风险

发现关键：需要一个"干净"的重写版

追问二：如何在没有源码的情况下实现？ #

这是一个技术难题。没有源码，怎么保证功能一致？

深入研究后发现 instructkr 团队使用了 OMX 框架：

  • OmX (Oh My Codex)：担任"总指挥官"，负责任务拆解、并行协调和架构验证
  • OmO (Oh My OpenCode)：担任"执行加速器"，负责代码快速实现

核心模式暴露：

  1. $team 模式：并行代码审查 + 多Agent协作
  2. $ralph 模式：持久执行循环 + 架构师级验证

再次追问：OMX 的调度机制是怎么工作的？ #

OmX 作为编排层，是如何让多个AI智能体协同工作的？

通过分析项目发现：

  • 任务分解：将自然语言目标拆解为原子任务
  • 角色分配：Architect、Coder、Reviewer、Tester 各司其职
  • 并行执行：同时运行多个智能体，大幅提升效率
  • 结果聚合：所有任务完成后，用静态分析工具进行最终验证

最终方案 #

1. 净室重写策略 #

  • 不包含任何原始代码
  • 只保留功能行为，不复制实现细节
  • 通过行为测试确保功能对等

2. OMX 工作流编排 #

用户需求 → OmX(任务分解) →并行执行→OmO(代码实现)
                            ↓
                        架构师验证 ↔ 失败反馈

3. 生产级特性 #

  • 交互式 REPL（增强的命令行界面）
  • LSP 客户端集成，完美适配 VS Code
  • 插件系统，支持功能扩展
  • 多模型切换（Claude、OpenAI、Gemini 等）

4. 惊人的发布数据 #

  • 2小时：50,000+ Stars
  • 24小时：100,000+ Stars
  • 总代码量：92.8% Rust，7.2% Python

总结 #

问题解决了吗？ #

是的。通过净室重写 + OMX 编排 + 多Agent协作，Claw Code 不仅规避了法律风险，还通过工程效率实现了现象级的 Star 数增长。

有什么局限性？ #

  1. 维护困难： Fork 功能被锁定，说明后续可能难以大规模分支
  2. 技术门槛： 基于 Rust 和 MCP 协议，对开发者有一定技术要求
  3. 依赖 Anthropic： 目前主要基于 Claude 模型，多模型支持还在完善中

还可以如何改进？ #

  1. 更好的文档：官方需要提供更详细的使用指南
  2. 社区治理：所有权转移后，如何建立可持续的开源治理模式？
  3. 商业化探索：企业版和安全隐私增强应该成为下一步重点

核心启示 #

Claw Code 的成功揭示了三个关键要素：

  1. 时机：在代码泄露的"黄金72小时"内抢占先机
  2. 技术：用工程化手段（OMX）替代简单复制
  3. 社区：通过"净室"方式响应开发者对安全和合规的需求

它的意义远不是一个"替代品"，而是展示了 AI Agent 框架的巨大潜力。未来，我们可能会看到更多基于类似理念的自主编程工具出现。
