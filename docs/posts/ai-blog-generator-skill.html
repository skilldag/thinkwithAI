AI Blog Generator #

从调研结果生成博客文章的工作流。

核心理念 #

问题驱动 + 深度展开 + 拆分/合并审核

以问题为导向，层层展开分析，最后审核是否需要拆分或合并。

工作流程 #

第一步：接收主题 #

用户输入要生成文章的主题/问题。

第二步：搜索调研结果 #

使用 cdp-scrape skill 搜索相关调研内容：

bash
# 查看所有已存储的调研文档
doc list

# 搜索相关内容
doc search 关键词

如果没有找到相关调研结果，提醒用户先去 deepseek-research 进行调研。

第三步：生成文章 #

文章结构根据实际内容灵活安排，通常包含：

  • 背景：具体场景和痛点
  • 分析：层层深入的核心讨论
  • 方案：从调研中提取的完整解决思路
  • 总结：局限性和改进空间

文章结构示例 #

markdown
---
title: 问题标题
description: 一句话描述解决的问题和方案
date: YYYY-MM-DD
---

# 问题标题

> 核心问题是什么？

## 背景

- 问题出现的场景
- 之前尝试过的方法及其效果

## 分析

### 第一层：表象

描述直接观察到的现象...

### 第二层：机制

深入探讨背后的原因...

### 第三层：本质

触及核心本质...

## 方案

完整的解决思路

## 总结

- 解决的问题
- 局限性
- 改进方向

文章章节数量根据内容复杂度调整，避免机械套用固定格式。

文件命名 #

  • 放在 docs/posts/ 目录下
  • 文件名使用 kebab-case
  • front matter 包含 title、description、date

第四步：审核拆分/合并 #

主文章生成后，评估是否需要拆分或合并：

情况操作
内容太多，一篇文章讲不清拆分成多篇，作为子文章
只有一个点，三言两语就说完合并到现有文章，作为补充
刚刚好，一篇文章讲一个点保持独立

判断标准：

  • 拆分：涉及 3+ 个独立子问题，或每个子问题都需要大量代码示例
  • 合并：核心内容 < 300 字，没有新的实践价值

拆分后的子文章作为主文章的 sidebar 嵌套。

第五步：更新配置 #

添加到 docs/.vitepress/config.ts 的 sidebar 配置中。

第六步：提交部署 #

bash
git add docs/posts/
git commit -m "feat: 添加文章《标题》"
git push origin gh-pages

关键要点 #

  • 问题驱动：从具体问题出发，不是泛泛而谈
  • 深度展开：层层深入，核心部分讲透
  • 拆分/合并审核：复杂内容拆成子文章，简单内容并入现有文章

依赖 Skills #

  • cdp-scrape: 搜索调研结果
  • deepseek-research: 调研工作流（调研由用户自行完成）

注意事项 #

  • 生成内容放到 docs/ 下面
  • 使用 VitePress 格式
  • 确保 sidebar 配置正确
  • 验证构建: pnpm build
