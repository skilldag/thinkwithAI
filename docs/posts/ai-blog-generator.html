AI Blog Generator Skill 设计思路 #

如何让 AI 生成的文章不是"编"出来的，而是真实记录问题解决过程？

问题背景 #

之前设计了 AI 生成博客的工作流，但在实际使用中遇到了问题：

  • 用户说"我要写一篇关于 React 的文章"，但这不是一个问题
  • 生成的文章像是功能说明文档，而不是问题解决记录
  • 内容太简单或太复杂，没有审核拆分机制

需要设计一个 Skill，来规范这个生成过程。

分析与方案 #

第一版设计：让用户随便输入主题 #

用户输入什么，我就生成什么。

结果：

  • 内容空洞，没有具体问题
  • 没有问题解决过程，直接给答案
  • 读者不知道这个方案是怎么得出的

问题出在哪里？ #

Q: 一篇好的技术文章应该是什么样的？

A: 应该记录"问题是怎么解决的"，而不是"这个功能是什么"

Q: 那生成之前需要确认什么？

A: 需要确认具体的问题，包含：场景、痛点、期望、尝试

Q: 内容太多或太少怎么办？

A: 需要审核机制，复杂的拆分，简单的合并

最终设计 #

设计 AI Blog Generator Skill，核心流程：

  1. 接收主题：用户输入要生成文章的主题
  2. 搜索调研：使用 cdp-scrape 搜索相关调研结果
  3. 生成文章：按问题-背景-分析-方案-总结的结构
  4. 审核拆分：复杂内容拆成子文章，简单内容并入现有文章
  5. 更新配置：添加到 VitePress sidebar
  6. 提交部署：推送到 gh-pages

文章模板：

  • 问题背景：具体场景和痛点
  • 分析：层层深入的核心讨论
  • 方案：从调研中提取的方案
  • 总结：局限性和改进空间

更新 Sidebar 配置 #

生成文章后，需要手动更新 docs/.vitepress/config.ts 中的 sidebar 配置。

父子关系设置 #

使用 items 数组创建父子结构：

typescript
sidebar: [
  {
    text: 'Posts',
    items: [
      { 
        text: '父文章标题', 
        link: '/posts/父文章路径',
        items: [
          { text: '子文章标题1', link: '/posts/子文章路径1' },
          { text: '子文章标题2', link: '/posts/子文章路径2' }
        ]
      }
    ]
  }
]

索引与文章名对齐要求 #

  1. sidebar 标题必须与文章 H1 标题一致

    • 打开生成的 .md 文件，第一行 # 后面的标题即为文章标题
    • 将此标题复制到 sidebar 的 text 字段

  2. 检查示例

typescript
// 文章文件：docs/posts/qwen3-coder-next-rtx3090.md
// 内容：# Qwen3-Coder-Next 在 RTX 3090 上的运行问题与优化指南
// 配置：
{ 
  text: 'Qwen3-Coder-Next 在 RTX 3090 上的运行问题与优化指南', 
  link: '/posts/qwen3-coder-next-rtx3090' 
}

  1. 常见错误

    • sidebar 写"本地大模型"，文章标题是"本地编码大模型实践指南" → 不对齐
    • 父分组名称与子文章标题无关 → 混乱

  2. 构建后验证

    • 运行 pnpm build 构建站点
    • 将 docs/.vitepress/dist/* 复制到 docs/
    • 提交推送后刷新页面检查

总结 #

  • 问题解决了吗？基本解决，生成的文章更有结构
  • 有什么局限性？调研过程依赖用户自行完成
  • 还可以如何改进？考虑集成自动调研功能
