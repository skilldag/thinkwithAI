如何引导大模型从根本上解决Bug #

当我们使用AI辅助编程时，一个核心挑战出现了：大模型往往只理解你提供的代码片段，却缺乏对整个代码库、业务逻辑和系统架构的完整认知。这导致它们常常给出表面看似合理、实则治标不治本的解决方案。

本文聚焦于如何让大模型真正理解代码和bug，从根本上解决问题，而非停留在表面修复。

1. 为什么大模型无法从根本上解决问题 #

1.1 表面修复 vs 根本解决 #

表面修复的典型表现：

  • ✅ 错误信息消失了
  • ✅ 程序能正常运行了
  • ❌ 但：漏洞的根基仍在，可能会在其他场景复现
  • ❌ 后遗症：引入了新的耦合、技术债务或隐藏bug

根本原因分析的正确结果：

  • ✅ 理解了为什么出现问题
  • ✅ 修复了问题的根源
  • ✅ 验证：修复后相关问题不再出现
  • ✅ 收益：提升了代码质量和可维护性

1.2 大模型的理解局限 #

大模型在处理bug时的常见问题：

局限1: 缺乏全局上下文

  • 只能看到提供的片段，不知道上下游依赖
  • 不理解业务逻辑和产品意图
  • 不了解历史演变和技术债务

局限2: 快速模式匹配

  • 识别到相似的错误模式就给出"标准答案"
  • 基于统计概率而非逻辑推理
  • 忽略了特定场景的特殊性

局限3: 缺少验证机制

  • 无法实际运行代码验证假设
  • 不了解副作用和边界条件
  • 给出的解决方案可能引入新问题

2. 构建代码的全局认知图谱 #

2.1 提供项目级上下文 #

错误的做法：只贴错误栈附近的代码

正确的做法：提供完整的上下文关键信息：

markdown
# 对大模型的提问模板

## 项目结构

project/ ├── core/ # 核心业务逻辑 ├── api/ # HTTP 接口层 ├── service/ # 服务层 ├── model/ # 数据模型 └── utils/ # 工具函数


## 相关文件
- 入口文件：main.py → 加载配置，初始化依赖
- 业务文件：user_service.py → 处理用户相关逻辑
- 数据文件：user_model.py → 定义用户数据结构
- 配置：config.yaml → 包含数据库连接等

## 数据流向
用户请求 → api/user.py → service/user_service.py → model/user_model.py → database

## 当前问题
错误发生在 service/user_service.py 的第 42 行，错误信息："user update failed"

2.2 提供历史上下文 #

markdown
## 近期改动历史
- commit 3d4f5a2: 添加了用户状态校验
- commit a1b2c3d: 修改了数据库连接池配置
- commit 7e8f9a0: 更新了 ORM 框架版本

## 可能相关的 Issue
- #234: 用户并发更新时的数据不一致问题
- #189: 数据库连接超时

2.3 业务领域解释 #

markdown
## 业务说明
- "用户状态"有3种：active（活跃）、inactive（非活跃）、suspended（暂停）
- 只有 active 用户可以执行操作 X
- suspended 用户无法执行任何操作，需要手动审核恢复
- 状态转换需要经过状态机校验

## 用户的意图
用户试图将一个 suspended 用户恢复到 active 状态，但操作失败了

给大模型的提示：如果不理解某个类/函数的作用，先不要尝试修复，先询问它的职责

3. 精准描述Bug的5个维度 #

3.1 错误现象（Symptom） #

错误描述：

✅ 好的描述：
用户访问 /api/user/123 时，50%几率返回 504 Gateway Timeout，
错误日志显示 "database connection timeout after 30s"

❌ 差的描述：
接口有时报错

3.2 触发条件（Trigger） #

提供触发问题的完整上下文：

markdown
必现条件：
1. 用户 ID > 10000（老用户）
2. 在北京时间每晚 8-10 点（高峰期）
3. 连续调用 /api/user 接口 3 次以上
4. 使用 PostgreSQL 数据库（而非 MySQL）

3.3 环境上下文（Context） #

markdown
- OS: Ubuntu 22.04
- 运行时：Python 3.11.2
- 框架：Django 4.2.3
- 数据库：PostgreSQL 15，连接池 20
- 部署：3个节点的 k8s 集群
- 负载均衡：Nginx，轮询策略

3.4 已有尝试（Trials） #

告诉AI你试过什么、结果如何（这非常重要）：

markdown
我已尝试：
- ❌ 重启数据库服务 → 问题依旧
- ✅ 临时增加连接池到 50 → 超时概率降低（但仍然存在）
- ❌ 检查数据库慢查询 → 未发现异常（查询都在 100ms 内）
- ❌ 检查网络延迟 → 数据库连接平均 2ms

提示：告诉AI什么没解决问题，比告诉它什么解决了更有价值

3.5 预期行为 vs 实际行为 #

python
# 当前行为（错误）
def get_user(user_id):
    user = db.query(f"SELECT * FROM users WHERE id={user_id}")
    return user  # 这里有时会返回 None

# 预期行为
def get_user(user_id):
    # 应该始终返回有效用户对象
    # 如果用户不存在，应抛出 UserNotFound 异常
    user = db.query(...)
    if not user:
        raise UserNotFound(f"User {user_id} not found")
    return user

关键：告诉AI"什么是对的"，帮助它理解业务正确性标准

4. 引导AI进行根本原因分析（RCA） #

4.1 使用"5个为什么"框架 #

不要直接问"怎么修复"，而是引导AI深入分析：

markdown
让我们用"5个为什么"分析这个问题：

问题：用户支付失败，错误 "payment timeout"

Q1：为什么支付超时？
→ 因为第三方支付接口 30 秒未响应

Q2：为什么第三方支付接口 30 秒未响应？
→ 因为我们的请求参数格式不正确，导致对方卡住

Q3：为什么请求参数格式不正确？
→ 因为在构造参数时，使用了旧版 API 的数据结构

Q4：为什么使用了旧版 API 的数据结构？
→ 因为上周升级 SDK 时没有更新参数映射逻辑

Q5：为什么升级 SDK 时没有更新参数映射逻辑？
→ 因为没有自动化测试覆盖支付流程，开发没发现问题

根本原因：缺少支付流程的自动化测试 + 升级流程不完善
症状修复：调整参数格式
根本解决：添加支付自动化测试 + 建立升级检查清单

提示：要求AI先分析问题，再讨论解决方案

4.2 强制AI思考所有可能性 #

markdown

1. 假设的依据
2. 如何验证这个假设
3. 如果是这个原因的修复方案

我的期望：
- 不要急于给最终答案
- 先列出可能性
- 让我们逐一分析

markdown

Possible Causes:
1. HTTP Handler 异常

请帮我：
1. 分析哪个原因最有可能
2. 如何快速验证（日志？调试？）
3. 最佳修复方案

4.4 让AI扮演不同角色 #

markdown
请你分别扮演以下角色，给我不同视角的分析：

1. 经验丰富的架构师：从系统设计角度看问题
2. 资深后端工程师：从代码质量角度看问题
3. DevOps 工程师：从基础设施角度看问题
4. 产品经理：从用户体验角度看问题

然后综合各角色的观点，给出根本原因分析。

4.5 要求AI提供证据链 #

markdown
不要直接告诉我修复方案，请提供：
1. **证据**：哪些日志、代码、数据支持你的判断？
2. **推理**：从证据到你的结论的逻辑链条
3. **验证**：修复后如何验证问题已解决
4. **回滚计划**：如果修复无效，如何回退

关键：让AI展示它是如何得出结论的，而不是只给结论

6. 应对常见类型Bug的AI协作策略 #

6.1 性能问题 #

markdown
❌ 错误提问：
"我的API很慢，如何优化？"

✅ 正确提问：
"我的用户列表接口响应时间从 200ms 增加到了 3s，请帮我分析：

1. 当前性能指标：
   - median: 200ms (3周前) -> 3000ms (现在)
   - p95: 500ms -> 8000ms
   - p99: 1000ms -> 15000ms

2. 调用链路：
   API → service → 3个数据库查询 + 2个外部API

3. 数据库查询（附上慢查询日志）：
   Query 1: SELECT * FROM users WHERE ... # 100ms
   Query 2: SELECT * FROM orders WHERE ... # 2500ms ← 怀疑是这个
   Query 3: SELECT * FROM roles WHERE ... # 50ms

4. 我怀疑是 N+1 查询问题，请帮我：
   - 验证是否存在 N+1
   - 如果是，给出具体修复方案
   - 如果不是，分析其他可能原因
"

6.2 并发竞争问题 #

markdown
问题：偶现的数据库 "duplicate key" 错误

✅ 正确提供给AI的信息：

1. 代码片段（展示竞态条件）：
```python
# user_service.py
def create_user(email):
    # 问题：先查询再插入，存在竞态窗口
    if not User.objects.filter(email=email).exists():
        # ← 另一个进程可能在这里插入相同 email
        User.objects.create(email=email) # ← 导致插入时 duplicate
  1. 出现频率：每天在 10000 次调用中出现 2-3 次
  2. 相关环境：有 5 个 worker 进程
  3. 表结构：
sql
CREATE TABLE users ( 
    email VARCHAR(255) UNIQUE
)

预期AI的思考过程：

  • 认识到这是典型的 "check-then-act" 竞态条件
  • 提出使用数据库唯一约束作为原子性保证
  • 建议使用 UPSERT 或数据库原生并发控制
  • 讨论异常处理和错误重试策略

### 6.3 逻辑错误问题

```markdown
❌ 表面修复（AI可能给出）：
"在 calculate_total 函数里加个 try-except，捕获异常返回 0"

✅ 引导AI思考根本问题：

问题：用户反馈购物车优惠券计算错误

初始线索：
```python
def calculate_total(items, coupon):
    subtotal = sum(item.price * item.qty for item in items)
    discount = 0
    
    if coupon.type == "percentage":
        discount = subtotal * coupon.value / 100
    elif coupon.type == "fixed":
        discount = coupon.value # ← 问题？
    
    return subtotal - discount

进一步提供：

  1. 错误案例：用户购买 50 元商品，使用 60 元优惠券，最终价格为 -10 元
  2. 业务规则：优惠券最多抵扣到 0，不允许负金额
  3. 预期结果：最终价格应该是 0

引导AI的提问： "分析一下是否正确实现了优惠券规则？如果没有，给出根本性的解决方案"

预期AI分析：

  • 识别逻辑缺陷：没有限制最小金额
  • 提出修复：price = max(subtotal - discount, 0)
  • 更重要的：讨论是否在数据库层也有验证（数据一致性）
  • 业务角度：优惠券规则应该在哪里定义？防止多处业务逻辑不一致

### 6.4 环境/配置相关Bug

```markdown
问题：在 staging 环境正常，production 报错

你需要给AI的：

1. 对比表格：

| 配置项 | Staging | Production |
|--------|---------|------------|
| DB Host | db-staging.internal | db-prod.internal |
| DB Timeout | 30s | 5s |
| Log Level | DEBUG | ERROR |
| Cache | Redis | Memcached |
| Workers | 2 | 8 |

2. 关键发现：
- 在 staging 从未出现超时
- 在 production 每 100 次调用出现 3 次超时
- 数据库监控显示查询耗时正常

3. 预期AI的推理：
- 发现配置差异：timeout 5s vs 30s
- 推论：可能是高并发下连接获取慢，5s 不够
- 观察线索：workers 8 vs 2
- 根本原因：连接池在高并发下紧张
- 解决方案：增加 timeout + 连接池大小

> **关键**：让AI先分析差异，再推导原因，而不是直接给答案

7. 验证AI解决方案的正确性 #

7.1 避免"表面修复"的陷阱 #

markdown
假设AI给出的解决方案：
"在用户查询数据库前加个 try-except，捕获所有异常并返回空结果"

验证清单：

**问题1：是否隐藏了其他问题？**
- ❌ 可能隐藏：数据库连接池耗尽
- ❌ 可能隐藏：SQL 注入攻击
- ❌ 可能隐藏：意外的数据损坏
- ✅ 建议：只捕获预期的异常类型

**问题2：是否引入新bug？**
- ❌ 用户看到空白页面，以为没有数据
- ❌ 日志中没有错误信息，无法排查
- ❌ 上游调用无法区分"无数据"和"查询失败"
- ✅ 正确做法：抛出清晰的业务异常

**问题3：是否违背业务规则？**
- ❌ 用户应该看到最新的数据，却看到缓存数据
- ❌ 金融交易中的数据缺失可能导致严重后果

**问题4：是否可测试？**
- ❌ 这个修复很难写单元测试验证
- ✅ 应该：提供可测试的、职责单一的解决方案

7.2 要求AI提供验证方案 #

markdown
请给出解决方案时，同时提供：

## 验证方案
1. **单元测试**：编写测试用例验证修复
2. **集成测试**：验证与上下游的协作
3. **手动测试步骤**：提供可操作的验证流程

## 边界条件测试
测试以下情况：
- [ ] 正常情况
- [ ] 空数据
- [ ] 最大值/最小值
- [ ] 并发场景
- [ ] 异常情况

## 风险评估
- 可能影响的其他模块
- 回滚复杂度
- 数据迁移需求（如果需要）

7.3 区分"症状修复"和"根本解决"的检查表 #

检查项症状修复根本解决
是否只动了出错的地方✅❌
是否修改了配置/依赖可能经常
是否需要修改多处代码❌✅
是否需要添加测试不需要必须
是否涉及架构调整否可能
是否消除了技术债务否是
错误是否会再次出现可能不太可能
是否需要文档更新不需要需要

只有全部回答"根本解决"，才是真正解决！

8. 负反馈案例：AI如何给出错误方案 #

案例：循环依赖问题 #

问题描述：

python
# user_service.py
from order_service import get_user_orders
# order_service.py
from user_service import get_user_info
# 出现循环导入错误

AI可能给出的表面修复：

python
# user_service.py
# 将导入移到函数内部
def some_function():
    from order_service import get_user_orders  # ← 延迟导入
    return get_user_orders(user_id)

问题分析：

  • ✅ 导入错误消失了
  • ❌ 根本原因未解决：架构设计有问题，两个模块应该解耦
  • ❌ 副作用：每次调用都导入，性能下降
  • ❌ 代码质量：不符合Python最佳实践，难以维护
