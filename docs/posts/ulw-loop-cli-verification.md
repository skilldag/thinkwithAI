# ULW Loop CLI 程序验证思路

> 如何让 AI Agent 真正"验收" CLI 程序的功能，而不是只检查代码是否编译通过？

## 问题背景

在开发 CLI 工具时，我们经常需要验证：

- 交互式流程是否正常工作（wizard、init、prompts）
- 各种参数组合是否正确处理
- 错误提示是否清晰准确
- 服务启动/停止是否正常

传统的测试方式是写单元测试或集成测试，但这些测试有几个问题：

1. **无法测试真正的交互式体验** - 测试框架很难模拟真实的 TUI 交互
2. **覆盖不到边界情况** - 很多 CLI 的细节问题只有在真正运行时才能发现
3. **验证不够直观** - 我们更关心"用户看到这个界面是什么样"

这个问题困扰了我很久，直到我找到了 **Ultrawork Loop + PTY 工具** 的组合方案。

## 探究过程

### 第一次尝试：写更多测试

我首先想到的是写更多的测试用例。编写了：

- 单元测试：测试各个函数
- 集成测试：测试命令行调用
- E2E 测试：使用 Playwright 模拟浏览器操作 TUI

但效果不好：

- 测试写了很多，真正的问题还是漏掉了
- 交互式流程很难用测试框架模拟
- 测试通过不代表用户使用没问题

### 追问：为什么测试无法覆盖？

我开始思考：为什么测试无法覆盖这些问题？

答案是：**测试是"代码级别"的验证，而 CLI 问题往往是"界面级别"的。**

用户使用 CLI 时，看到的是一个一个的"界面"（提示符、菜单、进度条），而测试只验证了代码逻辑，没有验证"界面"是否按预期工作。

### 第二次尝试：手工验证

于是我改为手工验证每个功能：

1. 运行 CLI 命令
2. 观察输出
3. 输入各种值
4. 检查结果

这种方法效果更好，但：

- 效率太低
- 难以重复
- 无法自动化

### 第三次尝试：PTY + Ultrawork Loop

最终我找到了解决方案：

1. **PTY (Pseudo-Terminal)** - 在 OpenCode 中可以启动交互式终端会话
2. **Ultrawork Loop** - 自动化的迭代验证循环，配合 Oracle 进行验收

核心思路是：**以界面为单位进行验证，而不是以命令为单位。**

## 最终方案

### 核心概念

#### 1. 什么是 PTY 工具？

PTY (Pseudo-Terminal) 工具可以：

| 工具 | 功能 |
|------|------|
| `pty_spawn` | 启动交互式终端会话 |
| `pty_read` | 读取终端输出 |
| `pty_write` | 发送输入到终端 |
| `pty_kill` | 终止会话 |

这些工具可以捕获**几乎所有**终端显示的内容：
- 文本输出
- ANSI 颜色
- ncurses 界面
- 交互式 prompts

#### 2. 什么是 Ultrawork Loop？

Ultrawork Loop (ulw-loop) 是增强版的 Ralph Loop：

| 特性 | Ralph Loop | Ultrawork Loop |
|------|------------|----------------|
| 触发命令 | `/ralph-loop` | `/ulw-loop` |
| 验证机制 | ❌ 无 | ✅ Oracle 验证 |
| 适用场景 | 简单重复任务 | 关键任务质量保证 |

关键区别：输出 `<promise>DONE</promise>` 后，还需要 Oracle 验证通过才算真正完成。

### 验证工作流

#### 阶段 1：界面探索

首先全面了解 CLI 的所有界面：

```bash
# 主帮助
pty_spawn(command="/path/to/cli", args=[], description="Get main help")
pty_read(id="pty_xxx")

# 带 --help
pty_spawn(command="/path/to/cli", args=["--help"], description="Get full help")
pty_read(id="pty_xxx")

# 子命令帮助
pty_spawn(command="/path/to/cli", args=["subcmd", "--help"], description="Get subcommand help")
```

从帮助信息中提取功能树：

```
CLI Root
├── init (初始化)
│   ├── 项目名输入
│   └── 模板选择
├── dev (开发服务器)
└── build (构建)
```

#### 阶段 2：以界面为单位验证

**关键思路**：不要只验证命令，要验证每个"界面"。

例如，对于 `init` 命令：

| 界面 | 触发条件 | 输入 | 预期输出 | 状态 |
|------|---------|------|---------|------|
| 欢迎菜单 | 运行 init | - | 显示"请输入项目名称" | ✅ |
| 项目名输入 | 显示欢迎菜单后 | "test" | 显示"请选择模板" | ✅ |
| 模板选择 | 输入项目名后 | "1" | 显示"正在创建..." | ✅ |

每个界面都要验证：
- 预期提示是否出现
- 输入后响应是否正确
- 是否有错误或异常

#### 阶段 3：Oracle 验证

使用 Ultrawork Loop 时，Oracle 可以：

1. **运行 CLI 命令** - 使用 pty_spawn
2. **观察输出** - 使用 pty_read
3. **发送输入** - 使用 pty_write
4. **验证行为** - 检查输出是否符合预期
5. **输出验收结果** - `<promise>VERIFIED</promise>`

Oracle 还可以使用 PTY 工具进行：
- 开发服务器启动验证
- 测试执行结果验证
- 错误处理验证

### 验证报告格式

验证完成后，生成结构化报告：

```markdown
# CLI 界面验证报告

## 基本信息
- CLI 路径: /path/to/cli
- 版本: 1.2.3

## 界面验证矩阵

### 功能: init

| 界面 | 输入 | 预期输出 | 实际输出 | 状态 |
|------|------|---------|---------|------|
| 欢迎菜单 | - | 请输入项目名称 | 请输入项目名称 | ✅ |
| 项目名输入 | test | 请选择模板 | 请选择模板 | ✅ |
| 模板选择 | 1 | 正在创建... | Error: xxx | ❌ |
```

## 总结

### 问题解决了吗？

是的。通过 Ultrawork Loop + PTY 工具：

1. ✅ 可以验证真正的交互式体验
2. ✅ 可以覆盖各种边界情况
3. ✅ 验证结果直观可见
4. ✅ 可以自动化重复验证

### 有什么局限性？

1. **只支持 CLI** - GUI 应用无法使用
2. **需要终端模拟器** - 某些特殊 TUI 可能不支持
3. **Oracle 成本** - 比普通验证更耗时

### 还可以如何改进？

1. **预定义验证模板** - 针对常见 CLI 类型（wizard、menu-driven、service）提供模板
2. **录制回放** - 记录验证过程，方便调试
3. **可视化报告** - 生成更直观的验证报告
